feat: harden workflow checkpoints

This PR makes workflow checkpointing safer and cleaner: oversized strings are truncated, circular references are stubbed, failures now record which node was active, and successful runs delete the checkpoint by default. The checkpoint JSON schema is also documented for operators.

  • Sanitizes checkpoint payloads before JSON serialization (10K string cap, circular refs).
  • Tracks the active node id so failure metadata identifies where the run broke.
  • Cleans up checkpoint files on success unless --no-checkpoint-cleanup is passed.
  • Documents the checkpoint schema and adds a long-range plan.md for remaining work.
  • DRS review reported 2 quality findings (1 HIGH, 1 MEDIUM) that should be addressed.

System overview

PR-agnostic

The workflow runner executes YAML-defined graphs of agents and actions. These notes describe the touched subsystem as it exists independent of this PR.

Workflow graph

Workflows are defined in YAML as nodes with needs dependencies. The runner resolves execution waves or control-flow segments and runs nodes concurrently within a wave.

Runtime context

During execution the runner maintains context.nodes (results), context.artifacts (outputs), and context.loop (loop control state). Nodes reference artifacts via template strings such as {{artifacts.diff}}.

Checkpoint files

When --resume is enabled, the runner writes a JSON checkpoint under .drs/checkpoints/<key>.json after every node. A later run can reload completed nodes instead of re-executing them.

Checkpoint safety

Checkpoints are generated runtime state and are gitignored. A resumed run rejects the checkpoint if the workflow name, key, or inputs differ from those stored in the file.

Data flow

How state moves
1

Node produces output

runSingleWorkflowNode returns a WorkflowNodeResult, which is stored in context.nodes and, for non-skipped nodes, copied into context.artifacts by recordNodeArtifact.

2

Checkpoint is written

saveWorkflowCheckpoint snapshots context.nodes, context.artifacts, and context.loop into JSON. After this PR, the snapshot passes through sanitizeCheckpointValue to cap strings and stub circular refs.

3

Failure path

If a node throws, the runner catches the rejection and writes a checkpoint with a failure object containing message, failedAt, and now nodeId (see risks for accuracy concerns).

4

Success path

On a successful run, cleanupWorkflowCheckpoint deletes the checkpoint file unless --no-checkpoint-cleanup is set or the workflow input checkpointCleanup is false.

5

Resume path

A subsequent --resume run calls loadWorkflowCheckpoint, validates workflow/key/inputs, then restoreWorkflowCheckpoint copies stored nodes and artifacts back into the live context.

Risks and questions

HIGH DRS finding · QUALITY Checkpoint string truncation corrupts resumed workflow state

src/cli/workflow.ts · sanitizeCheckpointValue / saveWorkflowCheckpoint

Strings longer than 10,000 characters are truncated before the checkpoint is written. On resume, restoreWorkflowCheckpoint treats the sanitized values as completed node results, so downstream nodes may consume incomplete diffs or review findings. Review artifacts and filesWithDiffs[].patch can easily exceed this limit.

Inspection area: Should the runner skip the write when the payload is oversized rather than mutating workflow state? Are there any production workflows where 10K is a routinely exceeded artifact size?

MEDIUM DRS finding · QUALITY Shared activeNodeId can record the wrong node in failure metadata

src/cli/workflow.ts · runWorkflow / runWorkflowDagSegment

Both the static wave executor and the DAG-segment executor assign executionContext.activeNodeId = nodeId inside concurrent Promise.allSettled callbacks. The failure-handling code reads this shared mutable field, so when a wave has multiple runnable nodes the last node to start can overwrite the id of the node that actually failed.

Inspection area: Capture the node id in the promise closure and propagate it with the rejection instead of relying on shared mutable state.

Reviewer question: cleanup default vs. operator expectations

The new default deletes checkpoints on success. Operators who previously expected to inspect a successful checkpoint now need --no-checkpoint-cleanup. Is this change documented prominently enough, and should the workflow input checkpointCleanup be mentioned in docs/WORKFLOWS.md?

Reviewer question: plan.md as committed planning doc

plan.md is a large new file with many unchecked items and cross-references to other PRs. Does it belong in the repo root or in docs/? How will it be kept in sync once merged?

const CHECKPOINT_MAX_STRING_LENGTH = 10_000;

function sanitizeCheckpointValue(value: unknown, seen: WeakSet<object> = new WeakSet()): unknown {
  if (typeof value === 'string') {
    return value.length > CHECKPOINT_MAX_STRING_LENGTH
      ? `${value.slice(0, CHECKPOINT_MAX_STRING_LENGTH)}…[truncated]`
      : value;
  }
  // ... circular reference handling ...
}

async function cleanupWorkflowCheckpoint(checkpoint) {
  if (!checkpoint) return;
  try { await unlink(checkpoint.path); }
  catch (error) { if (error.code !== 'ENOENT') throw error; }
}
