Four perspectives

1. System overview

The review artifact subsystem stores the output of a DRS review as a versioned JSON envelope. An envelope has metadata (id, kind, scope, timestamps) and a payload containing findings and a summary. Findings are individual issues with state, disposition, source, fingerprint, and timestamps. The workflow runner previously only created artifacts and reported status; this PR adds in-place mutation.

ReviewFinding

id, fingerprint, issue, state, disposition, source, createdAt, updatedAt

ReviewArtifactPayload

schemaVersion, reviewId, summary, findings, usage, metadata

WorkflowArtifactEnvelope

schemaVersion, kind, id, scope, payload, createdAt, updatedAt

2. Data flow
1

Review completes

create-review-artifact builds a payload from the review result. save-artifact wraps it in an envelope and writes it to .drs/artifacts/... as {id}.json and latest.json.

2

Fix agent runs

In the GitHub workflow, the fix-issues agent attempts repairs for findings at or above inputs.fixSeverity.

3

Mark findings attempted

mark-fix-attempted calls review-artifact-update-findings with state: attempted and disposition: partial. The runner reads the saved envelope, mutates matched findings, and rewrites the file.

4

Downstream decisions

fix-diff checks whether the working tree changed. If so, a fix PR is created. The updated artifact now reflects that the fix was only an attempt.

Leaves: src/lib/review-artifact.ts and src/lib/workflow-artifacts.ts are pure library code with no changed dependents other than the runner. Seam: the runner accepts either a raw payload or a saved envelope as the artifact input, so nodes can operate before or after save-artifact.

4. Operator / workflow action
1

Define a mutation node

Use review-artifact-update-findings with optional ids, fingerprints, or severity selectors plus state and/or disposition.

2

Add findings manually

review-artifact-add-finding takes a JSON issue value (or an artifact reference) and a source. It allocates the next F### id and updates the summary counts.

3

Convenience aliases

review-artifact-promote-finding sets disposition to confirmed; review-artifact-resolve-finding sets state and disposition to resolved. Both delegate to the same update function.

4

Persistence is automatic

If the input artifact is a saved envelope, the runner calls updateWorkflowArtifact and returns the new path and latestPath in the node output.

Risks and questions

Confirmed DRS findings from the review are labeled. The remaining items are reviewer questions, not new findings.

Severity selector uses exact case-sensitive match, breaking mark-fix-attempted DRS finding · HIGH
.pi/workflows/github-pr-review.yaml · line 218

inputs.fixSeverity is lowercase (e.g. "high") but findings store severity as uppercase ("HIGH"). selectorMatches compares with exact equality, so mark-fix-attempted will match zero findings and higher-severity findings are also skipped.

Review question: Should the selector support a minSeverity rank comparison, or should the workflow normalize case before passing the value?

Unwrapped JSON.parse in review-artifact-add-finding node DRS finding · MEDIUM
src/cli/workflow.ts · line 1512

getReviewIssueFromNode calls JSON.parse(issueName) directly when with.issue is not an artifact reference. A malformed string raises a raw SyntaxError instead of the workflow node error.

Review question: Wrap the parse in a try/catch so callers get a contextual error.

isReviewIssue type guard misses required agent field DRS finding · MEDIUM
src/cli/workflow.ts · line 1489

The guard checks severity, category, title, file, problem, and solution, but ReviewIssue.agent is also required. An issue without agent can pass validation and later break rendering or posting.

Review question: Add typeof issue.agent === 'string' to the type guard.

Additional inspection areas

  • Idempotency: If a workflow is re-run, will mark-fix-attempted re-apply attempted / partial to already-resolved findings? The selector has no state filter.
  • Fingerprint stability: New findings added by addReviewArtifactFinding use createIssueFingerprint; verify fingerprints do not collide with existing agent findings.
  • Envelope vs payload: The runner output differs when an envelope is present (returns envelope with path) vs when only a payload is passed (returns payload). Workflow authors should ensure downstream nodes consume the correct shape.
